# Paper Plan — ITP Short Paper

**Title (draft):** Generating Lean 4 Tactics from User Specifications: A Study of LLM-Assisted Metaprogramming Strategies

**Venue:** ITP (Interactive Theorem Proving), short paper, ~8 pages
**Status:** Draft

---

## Framing

> An empirical study of strategies for LLM-driven Lean 4 tactic generation,
> comparing specification quality, planning, and tool-augmented repair.

The central insight: generating a working tactic requires more than asking an LLM for code.
The right scaffolding — a formal spec, a milestone plan, and live LSP feedback — makes the
difference between a tactic that compiles and one that works.

---

## 2×2 Study Design

The `limit_auto` tactic is studied under four conditions:

|                    | No milestone plan | With milestone plan |
|--------------------|-------------------|---------------------|
| **No LSP tools**   | condition A       | condition B         |
| **With LSP tools** | condition C       | condition D (best)  |

- Condition A: one-shot API call, no plan, no compiler feedback (baseline)
- Condition B: milestone plan generated first, implemented via API without LSP
- Condition C: one-shot implementation using Claude Code with LSP interaction
- Condition D: milestone plan + Claude Code with LSP (incremental)

Note: condition A (`one_run.lean`) was not fully executed — it is empty.
This is itself a result: a raw one-shot with no scaffolding produces nothing usable.

## Section Outline

### 1. Introduction (~1 page)
- Motivation: tactics are reusable proof infrastructure; writing them requires deep
  Lean metaprogramming knowledge (TacticM monad, TSyntax, evalTactic, Mathlib APIs)
- Claim: LLMs can generate working tactics if given the right scaffolding
- Preview the 2×2 study and the pilot experiment

### 2. Background (~1 page)
- Lean 4 metaprogramming in one paragraph: tactic monad, syntax quotations, evalTactic
- What makes tactic generation hard: type-correctness, Mathlib API names (deprecated,
  wrong arity, definition-unfolding pitfalls), elaboration order subtleties
- Prior work: LLM proof generation (Copilot, LEGO-Prover, etc.) but NOT tactic generation
  — this gap is the contribution

### 3. Pilot: Intuitionistic Implication Tactic (~1 page)
- Context: testing LLM-generated metaprogramming on a small decidable fragment
- Approach 1 (iterative): generate examples, feed errors back, adapt — slow
- Approach 2 (spec-first): ask LLM to characterize the class in natural language,
  then generate — much more efficient
- Honest caveat: gas-bounded search, not mathematically complete
- Takeaway: the "characterize first" insight motivates the spec-first pipeline

### 4. The `limit_auto` Study (~3 pages)

#### 4.1 The tactic
- What it does: proves `Tendsto f F G` goals automatically
- Three pattern classes: continuity, exponential decay, reciprocal
- Why it is a good test case: requires Mathlib API knowledge + metaprogramming structure

#### 4.2 The spec
- Generated by the LLM from a one-line user request
- Includes: scope analysis, mathematical spec (3 classes), algorithm, test generation
- Key role: anchors all four conditions to the same target

#### 4.3 Results per condition
- Condition A: empty — confirms raw one-shot produces nothing
- Condition B: compiles, ~5 milestones, but several test failures (wrong lemma names,
  no LSP to verify)
- Condition C: one-shot, compiles with zero errors after LSP-guided repair; all tests pass
- Condition D: full 9-milestone evolution, zero errors, richer test suite

#### 4.4 Concrete failure modes fixed by LSP
- `whnf` unfolds `Filter.Tendsto` to its definition → pattern matching fails
- `TSyntax \`tactic` vs `TSyntax \`tactic.seq` type mismatch
- `simp` inside `exact ... (by simp)` causes logged errors not suppressible by try-catch
- Deprecated lemma names: `tendsto_inv₀_nhdsWithin_ne_zero` → `tendsto_inv₀_nhdsNE_zero`

### 5. Discussion (~0.5 page)
- The spec is the hardest step: a bad spec wastes all downstream effort
- LSP feedback is essential for API correctness — no model can memorize all Mathlib names
- The milestone plan provides error isolation and monotonic test growth
- Condition C (no plan + LSP) beats condition B (plan + no LSP): tool access matters more
  than planning alone

### 6. Related Work (~0.5 page)
- LLM proof search: Copilot for Lean, LEGO-Prover, Hypertree proof search
- Tactic synthesis: almost no prior work — this is the gap
- Program synthesis from natural language specs
- LLM-assisted metaprogramming in other languages (Rust macros, Template Haskell)

### 7. Conclusion + Future Work (~0.5 page)
- Short-term: more tactics, formal metrics (repair rounds, test coverage, compile time)
- Long-term: a benchmark suite of "tactics from specs" for the community

## Key Claims to Support with Evidence

1. Spec-first generation is better than raw generation (supported by condition A vs others)
2. LSP feedback fixes errors that LLMs cannot predict statically (Section 4.4 examples)
3. Milestone plans provide monotonic improvement (condition D milestone graph)
4. The approach generalizes beyond `limit_auto` (pilot + `decide_list_theory`)
