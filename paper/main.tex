\title{Generating Lean~4 Tactics from User Specifications:\\
       An Empirical Study of LLM-Assisted Metaprogramming Strategies}

\date{}

\begin{abstract}
We present an empirical study of strategies for generating Lean~4 tactics from
natural language specifications using large language models~(LLMs).
Rather than measuring whether an LLM can generate a tactic from a bare prompt,
we ask what \emph{scaffolding} makes tactic generation reliable: a formal
specification, a milestone plan, and live compiler feedback via an LSP server.
Our main experiment is a $2 \times 2$ study of the \texttt{limit\_auto} tactic,
which automatically proves \texttt{Tendsto~f~F~G} goals in Lean~4 / Mathlib,
comparing the presence or absence of a milestone plan with the presence or
absence of live LSP tool access.  We find that raw one-shot generation without
scaffolding produces no usable output, while the combination of milestone
planning and LSP feedback produces a working tactic with zero compiler errors
and a rich test suite after nine incremental milestones.  We characterise the
concrete classes of errors that only LSP feedback can fix---incorrect Mathlib
API names, deprecated lemma identifiers, elaboration-order pitfalls---and
discuss the implications for LLM-based metaprogramming pipelines.
\end{abstract}

%%% ============================================================
\section{Introduction}
%%% ============================================================

Interactive theorem provers such as Lean~4 have become important tools for
formalising mathematics and verifying software.  A key feature of these systems
is their extensibility through \emph{tactics}---procedures that automate proof
steps and can be reused across an entire development.  Writing a non-trivial
tactic requires deep knowledge of the \texttt{TacticM} monad, Lean's expression
representation, the Mathlib library API, and numerous elaboration subtleties
that are difficult to predict without a live compiler.

This paper investigates whether large language models can bridge this gap,
generating working Lean~4 tactics from informal user specifications.
Our main case study is a tactic we call \texttt{limit\_auto},
generated from the following user request (see \texttt{experiments/limit\_auto/spec.md}):
\begin{quote}
\textit{``A tactic that proves limit statements (tendsto goals) by analysing
the mathematical structure of the limit expression\ldots''}
\end{quote}
The resulting tactic automatically proves goals of the form \texttt{Tendsto~f~F~G}
by syntactically classifying the function~$f$ and selecting the appropriate Mathlib
lemma.  It requires non-trivial knowledge of filter notation, Mathlib
lemma names, and metaprogramming patterns---making it a good test case for our study.

Our original question was simply: \emph{can an LLM generate this tactic?}
Once we found that the answer depended critically on how the task was presented,
the question sharpened to \emph{which scaffolding strategies make generation successful}.
We compare four conditions in a $2 \times 2$ design:
\begin{itemize}
  \item the presence or absence of a formal \emph{milestone plan} generated
        before implementation; and
  \item the presence or absence of live \emph{LSP tool access} during
        implementation (real-time diagnostics, goal inspection, lemma search).
\end{itemize}

By \emph{LSP tools} we mean the Lean Language Server Protocol tools available inside
Claude Code: real-time compiler diagnostics (\texttt{lean\_diagnostic\_messages}),
proof-state inspection (\texttt{lean\_goal}), and natural-language Mathlib
search (\texttt{lean\_leansearch}, \texttt{lean\_loogle}).
These give the model live feedback from the Lean kernel without requiring it to
run the compiler manually.

The paper also describes a pilot experiment on intuitionistic implication
tactics that motivated the specification-first pipeline used in the main study.

\begin{itemize}
  \item A $2 \times 2$ empirical study of LLM tactic generation strategies.
  \item A characterisation of the error classes that LSP feedback uniquely resolves.
  \item A reusable pipeline: \textbf{spec} $\to$ \textbf{plan} $\to$
        \textbf{LSP-guided implementation}.
  \item The \texttt{limit\_auto} tactic as a concrete, verified artifact.
\end{itemize}

\section{Background}
%%% ============================================================

\subsection{Lean 4 Metaprogramming}

Lean~4 provides a unified metaprogramming framework in which tactics are
written in Lean itself.  A tactic operates in the \texttt{TacticM}~monad, which
wraps \texttt{MetaM} (for unification and type inference) and
\texttt{ElabM} (for elaboration).  The key data types are:

\begin{itemize}
  \item \textbf{\texttt{Expr}}: the internal representation of Lean terms,
        types, and proofs.  Includes \texttt{.app}, \texttt{.lam},
        \texttt{.forallE}, \texttt{.lit}, etc.
  \item \textbf{\texttt{MVarId}}: a metavariable standing for an open proof
        obligation (a ``goal'').
  \item \textbf{\texttt{Syntax}}: the raw parse tree, used to quote and
        splice tactic expressions.
\end{itemize}

A tactic is registered via
\lstinline{syntax (name := myTac) "my_tac" : tactic} and
\lstinline{@[tactic myTac] def evalMyTac : Tactic := fun _ => do ...}.
The body inspects the current goal with \texttt{getMainGoal},
reads its type with \texttt{getType},
and can run sub-tactics via \texttt{evalTactic}.

\subsection{What Makes Tactic Generation Hard}

Generating a working tactic with an LLM is harder than generating ordinary
code for several reasons:

\begin{enumerate}
  \item \textbf{Mathlib API names.}  Mathlib contains tens of thousands of
        lemmas with non-obvious, versioned names.  Names change between
        releases (deprecation), and no LLM can reliably memorise all current
        identifiers.

  \item \textbf{Elaboration order.}  Lean's elaborator processes terms in a
        specific order.  Writing \texttt{exact f (by tac)} can cause
        \texttt{tac} to run before metavariables from the surrounding
        unification are resolved, producing errors that are invisible to
        static analysis.

  \item \textbf{Definition unfolding.}  \texttt{whnf} (weak-head normal form)
        aggressively unfolds definitions.  Calling \texttt{whnf} on a
        \texttt{Tendsto~f~F~G} expression silently unfolds \texttt{Filter.Tendsto}
        to \texttt{F.map~f~$\leq$~G}, making subsequent pattern matching fail.

  \item \textbf{Syntax node kinds.}  A multi-step tactic sequence has kind
        \texttt{tactic.seq}, not \texttt{tactic}.  Helpers typed as
        \texttt{TSyntax `tactic} reject sequences, causing type errors at
        the call site rather than the source of the mistake.
\end{enumerate}

\subsection{Prior Work}

LLM-based theorem proving has received significant attention.
LEGO-Prover~\cite{xia2024legoprover} accumulates reusable proof skills across sessions.
LeanDojo / ReProver~\cite{yang2024leandojo} retrieval-augments a language model
for Lean proof generation.
Hypertree Proof Search~\cite{lample2022hypertree} uses MCTS guided by a learned policy.
More recent systems---DeepSeek-Prover~\cite{deepseek2024prover},
AlphaProof~\cite{alphaproof2024}, and commercial efforts such as
Harmonic~\cite{harmonic2024}---have pushed the frontier of automated theorem proving
using reinforcement learning and large-scale Lean interaction.
All of these target \emph{proof generation}: finding sequences of \emph{existing} tactics.
\emph{Tactic synthesis}---generating new, reusable tactic procedures from
specifications---has received almost no attention in the literature,
despite being a practically important problem for library development.
Our work targets this gap.

Program synthesis from natural language specifications has been studied
extensively~\cite{gulwani2017program}, but the domain of
Lean metaprogramming combines a highly specialised API with strong
type-correctness requirements that differ qualitatively from general
code generation.

\section{Pilot: Intuitionistic Implication Tactic}
%%% ============================================================

\comment{I don't know if it should be that much detailed this part, but you tell me.}

Before the main study, we ran a pilot experiment to understand what makes
LLM-driven metaprogramming succeed or fail.
The target was a tactic that proves the implication fragment of
intuitionistic propositional logic: goals of the form
$P_1 \to P_2 \to \cdots \to P_n \to P_i$ where the conclusion
is one of the hypotheses.

\paragraph{Iterative approach.}
The first approach was purely reactive: generate a tactic, feed compiler
errors back to the LLM, and repeat.  This converged slowly.  The LLM
corrected local errors but introduced new ones with each round, and
without a prior specification it was unclear when the tactic was
semantically correct rather than merely compiling.

\paragraph{Spec-first approach.}
The second approach asked the LLM to first produce a mathematical
characterisation of the goal class:
\[
  \mathcal{F} = \{\,\varphi \mid \varphi \equiv P_1 \to \cdots \to P_n \to Q,
  \ Q \in \{P_1, \ldots, P_n\}\,\}
\]
and to describe the proof algorithm before writing any code.
With this specification in hand, implementation was significantly more
directed: the LLM had a clear target and the specification served as
an invariant during repair rounds.

\paragraph{Outcome and caveats.}
The final tactic uses gas-bounded proof search over the hypothesis context.
It handles the target formula class correctly but is not mathematically
complete: there exist provable formulas in the fragment that it cannot prove
within the gas bound.  This honest limitation motivated the more rigorous
spec-first pipeline used for the main study, and foreshadowed the key insight:
\emph{the quality of the specification determines the quality of the implementation}.

\section{The \texttt{limit\_auto} Study}
%%% ============================================================

\subsection{The Tactic}
\label{sec:tactic}

The tactic arising from the user request above is called \texttt{limit\_auto}
by the specification it generated for itself (Section~\ref{sec:spec}).
It automatically proves goals of the form \texttt{Tendsto~f~F~G} by
syntactically classifying the function~$f$ and dispatching to the appropriate Mathlib lemma.
It targets three pattern classes:

\begin{description}
  \item[Class~1 (Continuity)]
        $f$ is a composition of functions provably continuous by
        \texttt{continuity} or \texttt{fun\_prop}.
        Proof method: \texttt{Continuous.tendsto'} with the filter pair
        $(\mathcal{N}(a),\, \mathcal{N}(f(a)))$.

  \item[Class~2 (Exponential decay)]
        $f$ syntactically mentions \texttt{Real.exp}.
        Key lemmas: \texttt{Real.tendsto\_exp\_neg\_atTop\_nhds\_zero}
        ($\texttt{Tendsto}\;(x \mapsto e^{-x})\;\texttt{atTop}\;(\mathcal{N}\,0)$),
        and \texttt{Real.tendsto\_exp\_atBot}.

  \item[Class~3 (Reciprocal)]
        $f = x \mapsto x^{-1}$ in a normed division ring.
        Key lemmas: \texttt{Filter.tendsto\_inv$\text{\_}$0\_nhdsNE\_zero}
        and \texttt{Filter.tendsto\_inv$\text{\_}$0\_cobounded'}.
\end{description}

Additionally, the tactic performs a safety check that rejects syntactically
indeterminate forms ($t/t$, $(t{-}a)/(t{-}a)$, $t/0$, $0/0$)
before attempting any proof.

\subsection{The Specification}
\label{sec:spec}

The specification was generated by Claude~Sonnet from the one-line user request:
\begin{quote}
\textit{``A tactic that proves limit statements (tendsto goals) by analysing
the mathematical structure of the limit expression\ldots''}
\end{quote}
Before writing any code, the model was asked to produce:
(1)~a \emph{scope analysis} identifying the most feasible interpretation
given Lean's constraints;
(2)~a \emph{mathematical specification} naming the three classes with their
filter-pair conditions;
(3)~a \emph{test generation algorithm} describing how to systematically
enumerate provable \texttt{Tendsto} goals ordered by difficulty.

This specification served as the shared anchor for all four conditions:
every implementation was required to realise the same semantic contract.
The spec is stored at \texttt{experiments/limit\_auto/spec.md}.

\subsection{Study Design}

We study four strategies for \emph{generating} \texttt{limit\_auto} (Table~\ref{tab:design}).

\begin{table}[h]
\centering
\caption{$2 \times 2$ study design.}
\label{tab:design}
\begin{tabular}{lcc}
\toprule
                    & \textbf{No milestone plan} & \textbf{With milestone plan} \\
\midrule
\textbf{No LSP}     & A: \textit{one\_run/} (empty)    & B: \textit{plan\_without\_lsp/} \\
\textbf{With LSP}   & C: \textit{one\_shot\_with\_lsp/} & D: \textit{plan\_with\_lsp/} (best) \\
\bottomrule
\end{tabular}
\end{table}

An important methodological distinction separates the conditions.
\textbf{Condition A} was run via the automated pipeline (\texttt{pipeline/main.py}):
the model receives the spec via an API call, and a self-correction loop
automatically feeds compiler errors back for up to a fixed number of repair rounds.
\textbf{Conditions B, C, and D} were run interactively in Claude Code, where a human
researcher prompted the model to implement the tactic from the specification and,
where applicable, fed back the plan and previous milestone files at each step.
This distinction matters for interpreting Condition A's failure:
it is ambiguous whether the result reflects a limitation of the automated pipeline
scaffolding, the model, or both.

\textbf{Condition A} (no plan, no LSP) is the baseline: a single automated API call
with the spec as the only context, followed by a self-correction loop.
\textbf{Condition B} (plan, no LSP) is a Claude Code session in which the model
is given the spec and a milestone plan, and asked to implement each milestone
in sequence, with the previous milestone file and plan fed back as context at
each step, but no live compiler access.
\textbf{Condition C} (no plan, LSP) is a one-shot Claude Code session with full
LSP tool access throughout.
\textbf{Condition D} (plan + LSP) is a Claude Code session combining a 9-milestone
plan with LSP feedback at each step; this is the main positive condition.

\subsection{Results per Condition}
\label{sec:results}

\paragraph{Condition A.}
The automated pipeline produced no usable output: the model either declined to
generate metaprogramming code or produced a file that failed to compile, and the
self-correction loop did not converge within the repair budget.
As noted above, this failure is ambiguous: it may reflect the limitations of the
automated pipeline scaffolding (no interactive context, fixed repair budget),
the model's inability to synthesize metaprogramming code from a bare spec, or both.
What is clear is that the combination of no planning, no LSP access, and no interactive
human guidance produced nothing useful, confirming that bare automated generation
is not a viable strategy for tactic synthesis.

\paragraph{Condition B.}
The milestone plan (five milestones) gave the model a structured decomposition
of the problem. The generated code compiled, but several tests failed at
runtime. The root cause was incorrect Mathlib lemma names: without a live
compiler, the model used names it ``knew'' from training data, including a
deprecated identifier
(\texttt{tendsto\_inv$\text{\_}$0\_nhdsWithin\_ne\_zero}, renamed to
\texttt{tendsto\_inv$\text{\_}$0\_nhdsNE\_zero}).
The plan helped with architecture but could not substitute for API verification.

\paragraph{Condition C.}
One-shot implementation with LSP access produced a working tactic with zero
compiler errors after iterative self-repair driven by \texttt{lean\_diagnostic\_messages}.
All tests pass.  The implementation is slightly less structured than
Condition~D (no explicit milestone decomposition) but is functionally correct.
This result shows that LSP access matters more than planning alone:
C (no plan, LSP) strictly dominates B (plan, no LSP).

\paragraph{Condition D.}
The nine-milestone plan with LSP feedback at each step produced the most
complete result: zero errors, the richest test suite, and the most readable
architecture with clearly separated helper functions, named pattern classes,
and a registered trace class (\texttt{limit\_auto}) for debugging.
The milestone plan provided \emph{error isolation}---bugs introduced in one
milestone were caught before they could propagate---and \emph{monotonic test growth}:
each milestone added new tests that were never broken by subsequent milestones.

\subsection{Concrete Failure Modes Fixed by LSP}
\label{sec:failures}

The LSP tools used in Condition~D include
\texttt{lean\_diagnostic\_messages} (compiler errors/warnings),
\texttt{lean\_goal} (proof state inspection),
\texttt{lean\_leansearch} (natural-language Mathlib search),
\texttt{lean\_loogle} (type-pattern search),
\texttt{lean\_multi\_attempt} (testing multiple tactics without editing),
and \texttt{lean\_local\_search} (local declaration search).
Together they caught four classes of errors that the LLM could not predict:

\paragraph{(1) \texttt{whnf} unfolds \texttt{Filter.Tendsto}.}
An early version called \texttt{whnf} on the goal before pattern-matching.
\texttt{whnf} aggressively unfolds \texttt{Filter.Tendsto} to its definition
$F.\texttt{map}\,f \leq G$, making \texttt{matchTendsto?} always return
\texttt{none}.  The tactic failed on every goal with the message
``goal is not of the form Tendsto~f~F~G''.
\texttt{lean\_goal} at the matching site revealed that the goal type had
already been unfolded.  Fix: remove the \texttt{whnf} call entirely and
match on \texttt{instantiateMVars ty} directly.

\paragraph{(2) \texttt{TSyntax `tactic} vs.\ \texttt{tactic.seq}.}
The helper \texttt{tryTac} was originally typed as
\texttt{tryTac : TSyntax `tactic $\to$ TacticM Bool}.
Multi-step sequences such as
\texttt{apply Continuous.tendsto' <;> first | continuity | norm\_num}
have syntax node kind \texttt{tactic.seq}, not \texttt{tactic},
causing a type error at every call site that used a compound tactic.
\texttt{lean\_diagnostic\_messages} identified the kind mismatch.
Fix: change the argument type to \texttt{Syntax}
(which \texttt{evalTactic} accepts directly).

\paragraph{(3) \texttt{simp} inside \texttt{exact~f~(by~simp)}.}
For continuity goals with numeric targets (e.g., $2+1=3$),
the natural proof is \texttt{exact Continuous.tendsto' (by continuity) \_ \_ (by simp)}.
However, when \texttt{simp} runs inside a term elaboration context and makes
no progress, it writes an error to the message log \emph{before} throwing an
exception---so the error persists even when \texttt{tryTac} catches it.
This produced spurious diagnostics even on successful proofs.
Fix: replace the term-mode form with
\texttt{apply Continuous.tendsto' <;> first | continuity | norm\_num | ring | rfl | simp},
which runs tactics in sequence on subgoals and avoids the elaboration issue.

\paragraph{(4) Deprecated lemma names.}
\texttt{lean\_leansearch} in Milestone~6 found the cobounded inverse lemma as
\texttt{Filter.tendsto\_inv$\text{\_}$0\_nhdsWithin\_ne\_zero}.  The compiler
immediately flagged a deprecation warning:
the current name is \texttt{Filter.tendsto\_inv$\text{\_}$0\_nhdsNE\_zero}.
Without LSP access (Condition~B), the model used the old name, causing
the reciprocal tests to fail at runtime.

\section{Discussion}
%%% ============================================================

\paragraph{The specification is the hardest step.}
All four conditions shared the same specification, yet only the implementations
with LSP access produced correct code.  This suggests that a bad specification
is fatal regardless of scaffolding, while a good specification is necessary but
not sufficient.  Investing effort in specification quality---scope analysis,
precise class definitions, provability conditions---is the highest-leverage
action in the pipeline.

\paragraph{LSP feedback is essential for API correctness.}
No LLM can reliably memorise the current state of Mathlib's tens of thousands
of lemma names, especially as names are deprecated and renamed across releases.
The results confirm this: Condition~B failed not from logical mistakes but from
stale API knowledge.  A live compiler acts as a ground truth that the model
cannot replicate internally.

\paragraph{Plans provide error isolation, not API knowledge.}
The milestone plan in Condition~D did not contribute to correct lemma names---
that was the role of \texttt{lean\_leansearch} and \texttt{lean\_loogle}.
What the plan contributed was \emph{modularity}: each milestone had a narrow
scope, so errors introduced at one step were caught before they could obscure
errors at later steps.  The plan also ensured that each milestone added new
tests, giving a concrete definition of ``done'' at each step.

\paragraph{Condition~C beats Condition~B.}
This ordering---one-shot with LSP access beats planned without LSP---is perhaps
the sharpest result of the study.  It implies that tool access to the live
compiler is more valuable than decomposing the problem into milestones, at
least for a tactic of this size.  For larger tactics, the balance may shift.

\section{Related Work}
%%% ============================================================

\paragraph{LLM-based proof generation.}
LEGO-Prover~\cite{xia2024legoprover} uses an LLM to generate and accumulate
reusable proof ``skills'' across a session.  Copilot for Lean~\cite{han2022proof}
and ReProver~\cite{yang2024leandojo} train models to predict the next tactic
in a proof.  Hypertree Proof Search~\cite{lample2022hypertree} frames proof
generation as MCTS with a learned policy.  These works target \emph{proof
generation}---finding sequences of \emph{existing} tactics---whereas our work
targets \emph{tactic synthesis}: generating new, reusable tactic procedures.

\paragraph{Program synthesis from natural language.}
The literature on program synthesis from specifications is extensive;
see~\cite{gulwani2017program} for a survey.  LLM-based code generation
(Codex~\cite{chen2021codex}, AlphaCode~\cite{li2022competition}) has
demonstrated strong performance on standard benchmarks.
Lean metaprogramming is qualitatively harder because correctness requires
type-level compliance with a complex API.

\paragraph{Tool-augmented LLMs.}
Using tools (compilers, search engines, REPL environments) to augment LLM
reasoning has become standard practice~\cite{schick2023toolformer}.
Our work is an instance of this paradigm applied to Lean metaprogramming,
with the Lean LSP server as the primary tool.
Closest in spirit is the use of REPL feedback in
AlphaCode~2~\cite{alphacode2} and in interactive proof
assistants~\cite{first2023baldur}.

\section{Conclusion}
%%% ============================================================

We have presented an empirical $2 \times 2$ study of LLM strategies for
Lean~4 tactic generation, using the \texttt{limit\_auto} tactic as a
test case.  The main findings are:
\begin{enumerate}
  \item Raw one-shot generation is not viable.
  \item A formal specification anchors all conditions and is the prerequisite
        for any successful generation.
  \item LSP feedback fixes errors (wrong API names, elaboration pitfalls)
        that LLMs cannot predict statically.
  \item Milestone planning and LSP feedback are complementary, and their
        combination produces the most complete result.
\end{enumerate}

\paragraph{Future work.}
As a preliminary step beyond \texttt{limit\_auto}, we ran the plan + LSP strategy
(Condition D) on a second tactic: \texttt{decide\_list\_theory}, which automatically
decides list-theoretic propositions (membership, equality, sublist).
The generation followed the same interactive Claude Code methodology as Condition D
(specification generated by the pipeline, milestone plan written by the researcher,
milestones implemented one by one with LSP access).
This provides a second data point suggesting the strategy is not specific to
\texttt{limit\_auto}, though a systematic multi-tactic study remains future work.
The repository also includes 12 draft specifications generated by the pipeline's
\texttt{analyze\_request} stage as candidates for future implementation experiments.

The most important next step is to run the same $2 \times 2$ study on additional
tactics to test whether the ordering of conditions is robust.  Longer-term, we hope
to build a benchmark suite of ``tactics from specifications'' that the community can
use to evaluate LLM-assisted metaprogramming systems, and to study whether the
approach generalises to tactics requiring deeper mathematical reasoning than
pattern matching.
